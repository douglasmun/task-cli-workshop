SOC Runbook — Investigating a Suspected C2 Beacon

A suspected command-and-control beacon is typically identified through network detection rules that fire on periodic outbound connections with consistent timing intervals, or through endpoint telemetry showing a process making repeated HTTPS connections to a domain with low historical prevalence. When such an alert fires, the analyst's goal is to determine whether the traffic represents legitimate software behavior or an active adversary implant.

Begin by profiling the destination. Query your threat intelligence platform and passive DNS sources for the domain or IP. Key indicators of malicious infrastructure include domain registration within the past 90 days, hosting on bulletproof or frequently abused ASNs, absence of prior appearance in any benign traffic corpus, and SSL certificates issued by automated CAs with short validity periods and no organizational validation. JARM fingerprinting, if available, can identify known C2 frameworks such as Cobalt Strike and Mythic with high confidence.

On the endpoint, identify the process responsible for the connection. Note the full executable path, parent process, and command-line arguments. Malware frequently masquerades as legitimate processes (T1036.005 — Masquerading: Match Legitimate Name or Location). Check the executable's digital signature and compare the file hash against threat intelligence feeds. Examine loaded modules and memory regions for injected code (T1055 — Process Injection). Extract any network configuration from memory if your EDR supports live memory analysis.

Escalation criteria that should trigger immediate incident declaration include confirmation of a known malicious hash or JARM fingerprint, evidence of process injection, presence of the beacon process in a startup persistence location (T1547), or any indication of lateral movement or credential access activity from the same host. Preserve a full memory image and network packet capture before taking any containment action. Document all findings with timestamps in your case management system before closing or escalating the ticket.

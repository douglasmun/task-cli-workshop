## Confirmed findings

### Correctness

**1. Priority stored off-by-one, making highest tier unreachable** — `priority.ts:18`
`(task as any).priority = priority - 1`. Priorities are 1-based (1 = highest),
so `--priority 1` persists as `0`; the top tier can never be stored. Line 25
also logs the *original* value, so the confirmation message contradicts what was
written to disk. Verified: `saveTasks`/`loadTasks` do no priority transform, so
the decremented value is what persists.

### Architecture

**2. Command writes to the store file directly instead of via `saveTasks()`** — `priority.ts:23`
`fs.writeFileSync(STORE_PATH, ...)` bypasses the store layer. CLAUDE.md: commands
must never touch the filesystem; all I/O goes through `src/store/tasks.ts`, which
already exports `saveTasks(tasks)`. Fix: call `saveTasks(tasks)`.

**3. Filesystem imports (`fs`, `path`) in a command module** — `priority.ts:1`
Direct FS dependencies forbidden in `src/commands/`. Both become dead imports
once the write routes through `saveTasks()`.

**4. `STORE_PATH` duplicated from the store layer** — `priority.ts:5`
Verbatim copy of the constant in `tasks.ts:5`. The on-disk path is a store-layer
concern; a correctly layered command has no knowledge of it.

### Tests

**5. New `setPriority` command has no working test coverage** — `priority.ts:8`
The only priority test, `__tests__/priority.bun.test.ts`, imports `bun:test` and
spawns `bun run ... add --priority high` — it exercises add/list with string
priorities, never `setPriority`. Under Jest (`ts-jest`, `testMatch **/__tests__/**/*.test.ts`)
the file matches the glob but `bun:test` is unresolvable (no bun types installed),
and `jest --passWithNoTests` swallows it. Net: zero coverage. A proper test
belongs at `__tests__/priority.test.ts` with `jest.mock('../src/store/tasks')`.

**6. Off-by-one priority decrement is untested** — `priority.ts:18`
No test asserts the persisted value, so the off-by-one is unverified.
(Skeptic caveat: a store-mock test can't catch it as long as the command bypasses
`saveTasks` — which is exactly finding #2. Fix the architecture first, then the
mock-based assertion works.)

**7. 'Task not found' error path is untested** — `priority.ts:11`
`console.error` + `process.exit(1)` on a missing id has no test asserting the
message, the `console.error` usage (CLAUDE.md-required), or the non-zero exit.

**8. Direct FS write path is untested and unverifiable via store mock** — `priority.ts:23`
Because the success path writes the file directly instead of via `saveTasks`, a
convention-following store-mock test can never observe persistence — a broken
save would go undetected. Routing through `saveTasks` restores the assertable seam.

## Refuted (verification killed it)

**Use of `any` type assertion violates strict-TypeScript convention** — REFUTED.
The finding claimed `(task as any).priority` attaches a field absent from the
`Task` interface. False: `src/types.ts:6` already defines `priority: TaskPriority`.
The skeptic correctly rejected it — a good live example of adversarial verify
removing a plausible-but-wrong finding.

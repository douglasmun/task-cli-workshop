0–5 min · frame

One loop under every demo

Why this slide: an expert doesn't prompt harder — they close the same loop deliberately, every time. Each of the six demos is just one of these four steps done well. Spotting the loop is the mental model to carry home.

Perceive→ Reason→ Act→ Observe↺
  • Perceive — CLAUDE.md, the repo, and the git diff give Claude grounding. (demo 2)
  • Reason — plan mode makes the thinking visible before any edit lands. (demo 2)
  • Act — skills, subagents, and workflows do the actual work. (demos 3–4)
  • Observe — hooks, tests, and adversarial review verify the result. (demos 4–5)
The gap between casual and expert use isn't better prompts — it's closing the loop on purpose.
5–15 min · memory + plan mode · highest ROI

Give it memory. Make it plan.

The two highest-ROI habits. CLAUDE.md is a file at your repo root that Claude reads every session — durable memory that survives context compaction. Plan mode forces Claude to propose its steps before touching any file.

$ bat CLAUDE.md   # show the project's memory
# it holds: the layered-architecture rule,
# and @docs/architecture.md (an auto-loaded ref)

Read once per session, no re-explaining. That layered rule is exactly what the review demo catches a violation of later.

# inside claude, press:
Shift+Tab   # cycle into plan mode

Add a --due <date> flag to the add
command, wired through the store.
# → Claude shows a PLAN first, edits nothing yet

Plan mode = Reason before Act. You approve the plan before a single line changes.

Memory + plan mode is the habit dabblers skip — and the one that pays back fastest.
15–30 min · skills + subagents · force multipliers

Package the repeatable. Delegate the rest.

Skills are reusable prompts saved in .claude/skills/ and invoked as /name — codify a workflow once, run it forever. Subagents are separate Claude instances you hand a scoped job, keeping your main context clean.

/skills                       # list this repo's skills
/add-feature list tasks by priority   # a skill taking $ARGUMENTS

# hand a read-only investigation to a subagent:
Use the Explore agent to map how a command
flows from CLI to the data file.
# → runs in the background; your main chat stays uncluttered
  • Explore is read-only — it can search and read but never edit, so it's safe to fan out. It inherits your session model (capped at Opus).
  • Subagents now run in the background by default, nest up to 5 deep, and can auto-commit / push / open a PR.
A skill turns a good prompt into a reusable command; a subagent keeps big jobs out of your main context.
30–45 min · the max-tier block · centerpiece

Deterministic multi-agent review

A workflow is a script (.claude/workflows/*.js) that fans out many subagents in a fixed order — same steps every run, unlike a normal chat. This one reviews the uncommitted priority.ts, which has three planted bugs, one per review dimension.

# IMPORTANT: /workflows is a VIEWER, not a launcher.
# You can't type "/workflows review-changes". Instead, ask:
Use a workflow to review the uncommitted changes.
# → approve the "Run a dynamic workflow?" dialog

/workflows      # now open the viewer to watch the live agent tree
  • It should surface all three: a correctness off-by-one, an architecture violation (writes the file directly, bypassing the store), and a missing test.
  • How it's fast: a pipeline with no barrier — a finding starts being verified while the next dimension is still being reviewed.
  • Why you trust it: each finding faces skeptic agents that try to refute it; only survivors are confirmed. (Contrast the model-driven /review-suite skill.)
  • Reasoning knobs: /effort high spends more thinking; /model opusplan = Opus plans, Sonnet executes.
Deterministic + adversarially verified = a review you can trust to run the same way every time. If it drags, fall back to docs/review-backup.md.
45–55 min · safety + governance · so it scales

Guardrails, not vibes

Hooks are shell commands the CLI runs automatically at defined moments — PostToolUse fires after Claude edits a file, PreToolUse fires before it runs a command (and can block it). This is how you make good behavior deterministic instead of hoping for it.

$ bat .claude/settings.json           # where hooks are wired up
$ bat .claude/hooks/format-file.sh     # auto-formats after every edit
$ bat .claude/hooks/block-dangerous-bash.sh  # blocks risky commands
  • Live proof: make any edit in a session → the format hook fires on its own, no prompt. That's the loop's Observe step, automated.
  • Permission model: Claude asks before risky actions. claude --permission-mode auto loosens that for trusted, sandboxed work.
Expert use scales on deterministic guardrails — the hooks catch what attention won't.

appendix · what the review should catch

The seeded scenario

How the centerpiece demo is rigged to work. The file src/commands/priority.ts is added but never committed — the workflow reviews the uncommitted diff, so the file must stay in git add -N state. It hides three deliberate defects, one per review dimension.

DimensionPlanted issue
Correctnesspriority - 1 off-by-one — --priority 1 becomes 0; top tier unreachable.
ArchitectureDirect fs.writeFileSync to the store file, bypassing saveTasks() — violates the layered rule in CLAUDE.md.
Test coverageNo __tests__/priority.test.ts for the new command.
 
# reset the stage between runs (regenerate the diff):
$ rm -f src/commands/priority.ts
$ cp workshop-seed.priority.ts.tmpl src/commands/priority.ts
$ git add -N src/commands/priority.ts
